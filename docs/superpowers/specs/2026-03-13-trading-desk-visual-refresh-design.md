# Trading Desk Visual Refresh Design

**Date:** 2026-03-13
**Status:** Approved for implementation planning

## Goal

将当前朴素、偏后台管理风格的前端界面，重构为一套更像 `本地量化研究终端` 的视觉系统。

这次重构覆盖主流程页面，目标是让产品整体呈现 `交易台轻仪表盘` 的专业感、效率感和桌面工具感，而不是只局部换颜色。

## Scope

This change covers:

- 整站主流程页面的统一视觉重设计
- 应用壳层、左侧导航、页面容器的统一升级
- 登录页、工作台、数据中心、策略中心、回测中心、结果页的整体视觉重做
- 全局设计变量、颜色系统、间距系统、卡片系统、表单和表格的统一
- 策略中心深色代码区和结果页分析区的重点强化

This change does not cover:

- 路由结构调整
- API 接口和业务行为修改
- Python 策略执行能力
- 新增图表数据能力
- 新增复杂前端状态机或权限系统
- 引入 Monaco 等重型代码编辑器

## Current State

当前前端存在以下问题：

- 整站页面普遍是轻量后台样式，产品识别度低
- 左侧导航只是功能入口，没有形成终端壳层气质
- 登录页过于朴素，缺少产品入口感
- 工作台、数据中心、回测中心和结果页在层级和节奏上比较弱
- 策略中心已经切到 Python 资产工作台，但视觉重量还不够
- 代码区、指标区、任务区缺少清晰的视觉主次

因此，这次不是局部修饰，而是一次整站风格重构。

## Chosen Visual Direction

用户已确认本次重构使用以下方向：

- 风格方向：`C. 交易台轻仪表盘`
- 覆盖范围：`主流程全覆盖`
- 信息密度：`专业紧凑`
- 策略代码区：`深色编辑器区`

### Design Intent

目标气质应接近：

- 轻量交易终端
- 本地研究工作台
- 专业量化桌面工具

而不是：

- 通用后台
- 互联网运营控制台
- 炫技大屏或金融宣传页

## Visual System

### Color System

页面基调采用：

- 冷静蓝灰作为结构骨架
- 橙色作为信号、动作和关键状态点缀
- 偏冷浅灰作为页面底色，避免大面积纯白

颜色职责应明确：

- 深蓝灰：导航、主要强调区、终端骨架
- 中性灰：边框、分隔、辅助文本
- 橙色：重要动作、关键指标、状态点
- 浅冷灰：页面背景和卡片背景层次

橙色只做少量强对比点缀，不能泛滥到整页按钮和装饰里。

### Typography

整体排版应偏紧凑：

- 一级标题稳定、硬朗，不追求夸张尺寸
- 模块标题明确，但不做大面积视觉喊话
- 正文和辅助说明控制在可快速扫读的范围
- 数字、状态、代码使用更高辨识度的字重和对比

### Surfaces

整站表面层级应由以下几类构成：

- `App background`
  - 冷灰底色，带轻微层次变化
- `Primary panel`
  - 主要工作区卡片，边界清晰、阴影克制
- `Secondary panel`
  - 摘要、状态、列表容器
- `Terminal panel`
  - 深色代码区或强对比数据区

### Controls

按钮、输入框、表格要更像桌面工具控件：

- 按钮不走过于圆润的消费级风格
- 输入框更紧凑，强调结构和可扫描性
- 表格加强行分隔与状态识别
- 标签和状态 pill 保持简短、锐利、可快速识别

## Layout Principles

### Shell

左侧导航要从“普通侧栏”升级为“终端壳层”：

- 更深的背景
- 更清楚的品牌和产品定位区
- 当前导航项更强高亮
- 可放入本地模式或会话状态等轻量系统信息

### Page Rhythm

整站页面应遵循统一节奏：

- 页面头部负责标题、页面定位和当前上下文
- 中部进入主要工作区
- 卡片与表格之间有明确分区，而不是随意堆叠

### Density

本次密度选择为 `专业紧凑`，意味着：

- 保留较高信息密度
- 留白服务于结构，而不是大面积空洞
- 重点在可扫描与效率，而非展示型海报布局

## Page Designs

### Login Page

登录页需要从“基础表单”升级为研究终端入口。

建议结构：

- 采用居中偏分栏或具有明显重心的构图
- 一侧表达产品定位、场景和核心能力
- 一侧展示紧凑的登录卡
- 背景加入轻量线性网格、信号线或市场终端氛围

目标是让用户一进入就感受到“正在进入一个研究终端”，而不是普通后台登录。

### Workbench

工作台应成为研究总览页。

建议包含：

- 顶部研究环境信息
- 关键摘要卡
- 最近回测
- 数据状态

重点不是放很多模块，而是让用户快速判断：

- 数据是否就绪
- 最近研究进度如何
- 下一步应该进入哪个工作区

### Data Center

数据中心应从“表单 + 表格”提升为“数据运行面板”。

结构建议：

- 数据源配置区
- 同步任务区
- 覆盖范围区

这些区块在视觉上要明显分开，并形成从配置到运行结果的阅读顺序。

### Strategy Center

策略中心是本次整站重构的视觉核心之一。

必须保留当前的 Python 策略工作台结构，但要提升成更像专业编辑工作区：

- 左侧策略列表更像资产目录
- 右侧编辑区更像终端编辑工作面
- 深色代码区成为视觉锚点
- 元信息字段紧凑化，减少“长表单”观感
- 保存 / 删除 / 恢复区域在视觉上更明确

### Backtest Center

回测中心应从简单发起页升级为任务发射台。

建议强化：

- 发起回测卡的动作感
- 历史任务列表的任务队列气质
- 当前仍使用旧回测逻辑这一点的承接表达

即便这一页后续会继续演进，也应该先具备像“任务工作区”的视觉骨架。

### Results Page

结果页是另一个视觉高点。

目标不是改变结果逻辑，而是把页面结构塑造成分析面板：

- 顶部总览
- 指标卡区
- 曲线主分析区
- AI 解读区
- 明细记录区

图表区和指标区应比当前更有主屏感，AI 面板应像辅助分析模块，而不是随意附着的次要内容。

## Component Boundaries

本次重构建议分三层推进：

- `App shell`
  - 应用整体壳层、背景、导航、内容容器
- `Foundation components`
  - 面板、按钮、输入框、表格、状态标签等共用视觉单元
- `Page-level compositions`
  - 各具体页面如何组合这些基础单元

这样可以避免每个页面单独写一套视觉逻辑，导致最后看起来像几套不同产品拼在一起。

## Technical Approach

### Theming

需要建立一套全局设计变量：

- 颜色变量
- 字体层级变量
- 间距变量
- 阴影和边框变量
- 圆角变量

页面和组件优先消费这些变量，而不是继续在各自文件里写散落的固定值。

### Controlled Rewrite

尽管这是 `方案 3`，实现方式仍应是 `可控重构`：

- 保持现有路由和组件职责
- 优先通过壳层和基础组件统一风格
- 对主流程页面做结构优化
- 不为样式重构引入新的业务耦合

## Risks And Guardrails

本次必须守住以下边界：

1. 不改路由结构
2. 不改 API 行为
3. 不改现有业务语义
4. 不因为视觉重构破坏现有主流程测试

需要特别注意：

- 策略中心虽然会显著变样，但 CRUD 行为不能回退
- 回测中心虽然会重做版式，但旧回测启动逻辑不能被误伤
- 结果页虽然要更像分析终端，但现有结果接口与 AI 面板集成保持不变

## Testing

本次主要是界面重构，因此测试重点是 `回归保护` 而不是新增复杂业务逻辑。

### Frontend

需要保留并通过现有前端测试，重点保护：

- App shell 导航
- 登录流程页面
- 数据中心、工作台、回测中心、结果页基本渲染
- 策略中心核心交互

### Visual Regression Strategy

若需要补充测试，应优先补结构型断言，例如：

- 关键标题仍存在
- 核心操作按钮仍可见
- 代码区和列表区仍按预期渲染

而不是写大量脆弱的 className 级测试。

## Success Criteria

- 整站主流程页面形成统一的 `交易台轻仪表盘` 视觉语言
- 左侧导航具备明确的终端壳层气质
- 登录页、策略中心、结果页成为最有辨识度的三个视觉入口
- 策略中心的深色代码区成为稳定视觉锚点
- 工作台、数据中心、回测中心不再显得像默认后台页面
- 现有前端与后端主流程测试继续通过
