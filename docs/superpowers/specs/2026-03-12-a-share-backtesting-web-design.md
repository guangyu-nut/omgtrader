# A 股量化回测网站设计文档

## 1. 文档目标

本文档定义项目 `V1` 的产品定位、MVP 边界、信息架构、核心业务对象、技术方案、风险与里程碑，用于后续 implementation plan 编写与研发执行。

本文档覆盖的对象是一个面向个人 A 股量化研究者、运行在 macOS 本地环境中的量化回测网站。`V1` 以研究闭环优先，不追求一开始做成完整量化平台。

## 2. 项目背景

个人量化研究者在做 A 股策略验证时，通常会遇到以下问题：

- 本地环境搭建复杂，数据、回测、结果展示分散在不同工具中
- 常见研究场景需要一定编程能力，不利于快速试错
- 分钟级成交模拟和策略结果复盘缺少统一界面
- 研究记录、参数版本和结果快照不易保存与复现

本项目的核心目标是把 `数据接入 -> 策略配置 -> 回测运行 -> 结果查看 -> 研究记录保存` 这条主链路产品化，并在此基础上加入 AI 辅助解读，提升个人研究效率。

## 3. 产品定位

### 3.1 一句话定位

一个面向个人 A 股量化研究者、macOS 本地优先、以股票池轮动回测为核心的研究网站。

### 3.2 核心价值

- 降低个人研究者从 0 到 1 开始做 A 股回测的门槛
- 用模板化策略配置替代早期编码门槛
- 用分钟线成交模拟提高回测结果可信度
- 用结果可视化和 AI 解读提升研究效率

### 3.3 目标用户

核心用户为个人 A 股量化研究者，具备以下特征：

- 理解基本交易与量化概念
- 主要在本机进行研究，不强依赖多人协作
- 更偏好“可配置模板 + 快速试验”，而非一开始编写完整 Python 策略
- 有查看收益曲线、回撤、调仓记录、成交明细、参数快照的实际需求

## 4. 产品目标与非目标

### 4.1 V1 产品目标

`V1` 需要完成以下目标：

1. 用户可以在 macOS 本地一键启动系统并登录
2. 用户可以配置数据源并拉取 A 股日线和分钟线数据
3. 用户可以配置股票池轮动模板策略并发起回测
4. 回测支持日级调仓、分钟线成交模拟与增强版 K 线撮合
5. 用户可以查看结果报表、明细记录并保存策略与历史回测
6. 用户可以使用 AI 查看结果解释与参数建议

### 4.2 明确不在 V1 范围内的内容

以下能力不进入 `V1`：

- 因子分析完整工作台
- 因子挖掘
- 多租户、团队协作、复杂权限系统
- 云端部署与分布式任务集群
- 任意 Python 策略编写能力
- 逐笔或盘口级撮合
- 实盘交易接入
- 通用拖拽式策略编排器

## 5. MVP 范围

### 5.1 MVP 核心问题

`MVP` 只解决一个核心问题：

用户能否在本机快速拿到 A 股数据，配置股票池轮动模板策略，用分钟线做成交模拟，得到可信的回测结果，并把研究过程保存下来。

### 5.2 MVP 功能清单

#### 账户与登录

- 本地轻登录
- 单机本地账户密码登录
- 登录态管理与退出登录

#### 行情数据管理

- 配置免费数据源与 token 数据源
- 拉取 A 股基础证券信息
- 拉取交易日历
- 拉取指数基准
- 拉取并更新日线行情
- 拉取并更新分钟线行情
- 展示数据覆盖区间、最近更新时间、同步任务状态

#### 策略配置

- 创建股票池轮动模板策略
- `V1` 最少提供 1 个基线模板策略：按可配置排序指标选出 `Top N` 股票并等权持仓
- 配置股票池
- 股票池输入方式仅支持手工选股、CSV 导入和常见指数成分股导入
- 配置调仓频率为日级
- 配置排序规则、持仓数、基准
- 配置手续费、滑点、涨跌停限制、成交限制
- 保存策略模板和具体参数实例

#### 回测运行

- 从策略配置生成回测任务
- 固化回测参数快照
- 调仓信号按日线级别生成，并在下一交易日的分钟窗口内执行成交模拟
- 使用分钟线执行成交模拟
- 生成回测日志、错误信息和运行状态

#### 结果展示

- 净值曲线
- 基准对比
- 收益、回撤、夏普、胜率、换手等指标
- 调仓记录
- 订单记录
- 成交记录
- 持仓变化
- 参数快照

#### 策略与历史记录保存

- 保存策略模板
- 保存股票池配置
- 保存回测任务
- 保存回测结果快照
- 基于历史记录再次运行

#### AI 辅助

- 指标解释
- 回测表现总结
- 两次结果差异总结
- 参数调优建议

### 5.3 MVP 成功标准

以下四项全部满足时，视为 `MVP` 达标：

1. 系统可在 macOS 本地启动并完成登录
2. A 股日线与分钟线可稳定入库并供回测读取
3. 股票池轮动模板策略可成功回测并生成可信结果
4. 用户可保存研究资产并查看 AI 辅助解读

## 6. 用户场景与核心流程

### 6.1 核心场景

- 首次使用者配置数据源并同步历史数据
- 研究者创建股票池轮动策略并反复调整参数
- 用户查看回测收益曲线、明细记录与回撤表现
- 用户对比不同参数组合的回测结果
- 用户在查看结果后用 AI 获取文字摘要和调优建议

### 6.2 主流程

系统主流程定义如下：

1. 用户登录系统
2. 用户在数据中心配置数据源
3. 用户拉取基础证券信息、交易日历、指数、日线、分钟线
4. 用户在策略中心创建股票池轮动模板策略
5. 用户配置股票池、调仓参数、选股排序规则、执行成本参数
6. 用户在回测中心发起回测任务
7. 系统生成回测快照并执行回测
8. 用户在结果中心查看曲线、指标、明细和 AI 解读
9. 用户保存策略与回测结果，并基于历史配置继续试验

## 7. 信息架构

`V1` 采用 6 个一级模块：

1. 登录页
2. 工作台
3. 数据中心
4. 策略中心
5. 回测中心
6. 结果中心

### 7.1 模块职责

#### 登录页

- 账户登录
- 登录态初始化

#### 工作台

- 最近回测
- 最近策略
- 数据同步状态摘要
- AI 研究摘要入口

#### 数据中心

- 数据源配置
- 数据同步任务管理
- 数据覆盖区间与更新时间查看
- 缺失与异常提示

#### 策略中心

- 创建和编辑参数化策略
- 配置股票池、基准与调仓参数
- 保存策略模板与实例

#### 回测中心

- 发起回测
- 查看回测任务列表
- 查看运行状态与错误日志

#### 结果中心

- 查看图表、指标与明细
- 查看参数快照
- 调用 AI 辅助解释
- 再次运行

### 7.2 页面设计原则

- 围绕研究工作流组织页面，而非后台管理菜单
- 策略配置页和结果页优先保证可用性
- 数据管理页强调数据是否就绪，而非暴露底层细节
- AI 作为结果页的辅助能力存在，不单独作为主模块

## 8. 核心业务对象

### 8.1 用户与会话

- `User`
  - 本地账户
  - 用户名、密码哈希、创建时间、最近登录时间
- `Session`
  - 本地登录态和会话信息

### 8.2 数据资产

- `Symbol`
  - 股票基础信息
- `IndexSymbol`
  - 指数基准信息
- `TradingCalendar`
  - 交易日历
- `BarDaily`
  - 日线行情
- `BarMinute`
  - 分钟线行情
- `DataSourceConfig`
  - 数据源配置与认证信息
- `DataSyncTask`
  - 同步任务记录
- `DataCoverage`
  - 数据覆盖区间、缺失状态、更新时间

### 8.3 策略资产

- `StrategyTemplate`
  - 策略模板类型
- `StrategyInstance`
  - 用户保存的策略配置
- `StockPool`
  - 股票池定义
- `RebalanceRule`
  - 调仓频率、调仓时间、持仓数、排序方式
- `ExecutionConfig`
  - 手续费、滑点、涨跌停约束、成交限制
- `BenchmarkConfig`
  - 基准配置
- `AiPromptProfile`
  - AI 解读相关配置

### 8.4 回测任务

- `BacktestJob`
  - 一次回测任务
- `BacktestSnapshot`
  - 回测启动时冻结的策略、股票池、执行参数、数据版本快照
- `BacktestLog`
  - 运行日志、告警和异常记录

### 8.5 回测结果

- `PortfolioCurve`
  - 净值曲线
- `PerformanceMetrics`
  - 汇总指标
- `RebalanceRecord`
  - 调仓记录
- `OrderRecord`
  - 订单记录
- `TradeRecord`
  - 成交记录
- `PositionRecord`
  - 持仓变化
- `AiInsight`
  - AI 输出的解释和建议

### 8.6 核心边界原则

- 原始行情数据与同步任务记录分开
- 策略模板与具体配置实例分开
- 股票池配置与回测任务分开
- 回测任务必须固化快照，不能依赖当前最新配置
- 原始回测结果与 AI 衍生内容分开

## 9. 技术方案

### 9.1 总体架构

`V1` 采用本地单体应用加 Web 界面的架构，分为以下五层：

- 前端展示层
- API 服务层
- 回测引擎层
- 数据访问层
- 外部集成层

该结构优先满足本地部署简单、模块边界清晰、后续可平滑升级的要求。

### 9.2 前端方案

- `HTML5 + CSS3 + JavaScript`
- `Vue 3`
- `ECharts`

前端负责：

- 登录与工作台展示
- 数据源配置与数据状态展示
- 策略表单配置
- 回测任务状态展示
- 结果图表与明细表格
- AI 输出展示

选择 `Vue 3` 的原因：

- 页面存在较多状态型表单和交互
- 结果页面需要图表、表格和参数视图之间切换
- 纯原生 JS 在维护成本和状态管理上更容易失控

### 9.3 后端方案

- `Python 3.11+`
- `FastAPI`
- `SQLAlchemy` 或 `SQLModel`
- `Pydantic`
- `Uvicorn`

后端模块建议拆分为：

- `auth`
- `market_data`
- `strategies`
- `backtests`
- `results`
- `ai_assistant`
- `system_settings`

### 9.4 数据库方案

- 默认数据库：`SQLite`
- 兼容扩展：`MySQL 8.0+`

选择理由：

- 本地零部署成本低
- 适合单机优先场景
- 能满足 `V1` 的功能验证目标

性能边界：

- 对个人研究场景足够
- 大规模分钟线数据与批量任务性能不是 `V1` 优化重点

### 9.5 回测引擎方案

`V1` 采用自研轻量回测内核，不深度依赖第三方大框架。

引擎职责包括：

- 解析参数化策略模板
- 执行股票池轮动逻辑
- 根据日级调仓规则在日线收盘后生成调仓事件
- 默认在下一交易日开盘后的分钟窗口内进行成交模拟
- 使用分钟线进行成交模拟
- 支持手续费、滑点、涨跌停、基础成交限制
- 计算绩效指标并写入结果存储

不采用第三方框架作为核心基座的原因：

- 需要掌控产品模型与前后端交互方式
- 避免被既有框架的数据结构和运行范式强绑定
- 便于后续继续扩展策略模板与任务管理方式

### 9.6 数据接入方案

数据接入采用双层适配模式：

- 免费源适配器
- token 数据源适配器

适配器抽象为以下能力：

- `symbol provider`
- `daily bar provider`
- `minute bar provider`
- `calendar provider`
- `index provider`

这样可以在不影响策略与回测逻辑的前提下替换或新增数据源。

### 9.7 AI 集成方案

AI 作为辅助层存在，不进入回测主计算链。

支持能力：

- 指标解释
- 回测总结
- 结果差异对比
- 参数调优建议

不支持能力：

- 直接生成可执行策略并自动运行
- 修改核心回测结果
- 参与回测计算逻辑

建议将 AI 调用封装为独立服务接口，支持 OpenAI 或兼容 API，并保留模型参数与提示模板配置入口。

## 10. 数据流与系统交互

### 10.1 数据同步流

1. 用户配置数据源
2. 系统创建数据同步任务
3. 数据源适配器拉取证券、指数、交易日历、日线、分钟线
4. 系统清洗并写入数据库
5. 系统更新数据覆盖信息和同步状态

### 10.2 回测执行流

1. 用户选择策略配置并发起回测
2. 系统生成回测快照
3. 回测引擎读取快照和所需行情数据
4. 引擎按日级调仓规则生成调仓信号
5. 引擎在下一交易日的分钟窗口内使用分钟线模拟撮合
6. 引擎输出曲线、指标和明细记录
7. 系统写入结果并提供可视化查询

### 10.3 AI 解读流

1. 用户在结果页触发 AI 解读
2. 系统整理结果摘要、关键指标和参数快照
3. AI 服务生成说明与建议
4. 系统保存 AI 输出供复查

## 11. 错误处理与边界条件

以下错误场景必须在设计阶段明确：

### 11.1 数据层错误

- 数据源连接失败
- token 无效或过期
- 某股票数据缺失
- 分钟线区间不完整
- 交易日历与行情不一致
- 某股票停牌或开盘后分钟线无可成交数据

处理原则：

- 同步任务保留失败原因
- 数据中心明确展示覆盖状态和异常信息
- 回测前进行数据完整性校验

### 11.2 策略配置错误

- 股票池为空
- 调仓参数非法
- 手续费或滑点超出合理范围
- 基准未配置

处理原则：

- 前端表单即时校验
- 后端再次校验并返回可读错误信息

### 11.3 回测执行错误

- 快照创建失败
- 所需行情不存在
- 某次撮合无法成交
- 指标计算异常

处理原则：

- 回测任务状态可追踪
- 错误写入任务日志
- 前端允许用户查看失败原因并重新运行

### 11.4 AI 辅助错误

- API 调用失败
- 模型不可用
- 响应超时

处理原则：

- AI 失败不影响原始回测结果查看
- 明确提示 AI 服务不可用
- 保留重试入口

## 12. 测试与验证原则

虽然当前阶段不写代码，但 `V1` 后续实现应围绕以下验证维度展开：

### 12.1 产品级验证

- 新用户是否能在本地成功启动并登录
- 是否能完成一次完整数据同步
- 是否能从策略配置走到回测结果查看
- 是否能保存并复用历史策略和结果

### 12.2 数据级验证

- 数据源切换是否影响统一读取接口
- 日线和分钟线是否具备可追踪的数据覆盖信息
- 数据缺失和异常是否能被识别与暴露

### 12.3 引擎级验证

- 日级调仓逻辑是否符合配置
- 分钟线成交模拟是否符合撮合规则
- 手续费、滑点、涨跌停和成交限制是否正确生效
- 回测快照是否可复现历史结果

### 12.4 结果级验证

- 关键绩效指标计算是否正确
- 图表与明细是否来自同一份结果数据
- AI 输出是否与原始结果隔离

## 14. 主要风险与控制建议

### 风险 1：分钟线需求导致 MVP 失控

控制建议：

- 只做日级调仓加分钟线成交模拟
- 不做逐笔撮合
- 不做任意分钟事件驱动策略

### 风险 2：策略抽象过度

控制建议：

- 仅支持少量模板化轮动策略
- 不做通用 DSL
- 不做复杂拖拽编排

### 风险 3：数据源质量影响体验

控制建议：

- 免费源与 token 源双层方案
- 数据覆盖与更新时间显式展示
- 回测记录保存数据快照和来源

### 风险 4：AI 抢占主线

控制建议：

- AI 只做辅助解释
- 不进入回测主链路
- 所有 AI 输出可单独失败，不影响主功能

## 15. 最终结论

`V1` 不是一个大而全的量化平台，而是一个围绕个人 A 股量化研究者最常用工作流设计的研究闭环产品。

其最终 `MVP` 可以概括为：

用户在 macOS 本地登录系统后，可以接入 A 股数据源，管理日线与分钟线数据，配置股票池轮动模板策略，按日级调仓并基于分钟线做成交模拟回测，查看结果报表，保存策略与记录，并获得 AI 辅助解读。
